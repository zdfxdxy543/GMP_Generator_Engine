# 控制程序自动生成方案备份（2026-04-07）

## 背景
目标是让大模型根据“控制结构设计”生成控制程序主体逻辑，再由本地程序自动补齐声明与定义，最终得到可编译、可验证的控制代码。

## 当前共识
1. 模块必须使用指定 ID。
2. 控制结构输入只描述结构与参数，不要求完整 C 语法。
3. 本地程序按模块 ID 在知识库中查元数据并补全代码。
4. 大模型主要负责函数体主体（流程、调用顺序、控制逻辑），本地程序负责声明/定义、类型一致性和工程化修复。
5. 工作流采用“生成 -> 补全 -> 编译检查 -> 回灌修复”的闭环。

## 推荐边界分工
### 大模型负责
- init/step/fault 等函数体主体
- 模块连接顺序与调用序
- 参数使用与模式分支逻辑

### 本地程序负责
- struct/type/extern/函数原型补齐
- 头文件依赖与命名一致性
- 硬件接口映射（ADC/PWM/编码器）
- 缺失变量、参数默认值、宏定义兜底

## 核心风险
1. 知识库中 module_id 不唯一会导致补全冲突。
2. API 只有签名没有语义时，模型可能错连输入输出。
3. 缺少调度语义（fast/slow loop）会让生成代码可运行性下降。

## 解决策略
1. 增加 canonical_id（全局唯一）。
2. 增加 api_contract（参数语义、必填项、默认值）。
3. 增加 schedule_hint（init/fast/slow/fault）。
4. 让控制结构输入显式包含 links（模块间连接）和 params。

## 最小可行版本（MVP）
1. 使用控制结构 JSON 输入（见 schema 文件）。
2. 用 module_id 查 knowledge_base/component_modules_kb.json。
3. 生成 ctl_main.c 主体函数草稿。
4. 本地补齐声明定义后进行编译与静态检查。
